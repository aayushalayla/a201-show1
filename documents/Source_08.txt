Reply:
The majority of my projects are from doing a tutorial to learn some new tech, but then I take it further and keep finding new things to add to it. As I learn new things, I go back and find ways to implement it in old projects or update them.

My biggest project is one I thought up myself and started when learning React. Then I went back and added Redux. It uses a couple different APIs, SCSS, and a few other things. In the GitHub README, I document all the changes I’ve made along the way, as well as future plans like login, profiles, favorite lists, etc.

I implemented CI/CD in all my projects, and they are all deployed so someone can see them in action. If the project has an admin or user account, someone can either sign up or use the demo user/password.

When I talk about my projects, I start with what it was if I started with a tutorial, then discuss how I kept adding new features and improving it as I learned new tech, plus whatever new thing I’m interested in learning and plan to add.

I’m able to talk with excitement over my projects, talk in depth about what I can do, what issues I had and how I fixed them. There are a ton of things in my projects that interviewers have talked with me about that I can do better, which is useful because I can take that into the next interview and talk about what I learned, how I handle criticism and suggestions, and that I’m still open to learning new things.

I’m not a dedicated recruiter, but I have spent a good deal of time recruiting over the years. The only time I actually look at projects on a resume is to prepare for the interview.
