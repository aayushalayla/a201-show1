I once got hired for a Go programming job because I'd recently pushed an open-source Go program onto my GitHub account.

A CEO asked their admin assistant to search GitHub for recent projects with a set of parameters and return some candidates, and I was among them.

The actual project itself may not matter, but I'd avoid calculators, blogs, etc. -- anything that "everyone" does when they're learning. If you have something you actually use, or even better, that others actually use, that would be awesome.

Otherwise, you've got to just hope that your project is relevant to the company you're interviewing with.

Or, you could try to create a software product and go into business for yourself.

I uploaded my HackerRank, LeetCode, Codeforces, etc. solutions to GitHub.

When asked how the repo was organized, they said they divided solutions by platform and put each solution in its own script file. Their naming convention included the platform, problem number, problem name, and algorithm or data structure used. They also used subfolders for language, difficulty, problem number, and unsolved problems when needed.

They later explained that they did not put the repo on their resume. Before a user interview, the interviewer opened their GitHub, found the repo, browsed it, picked three random problems, and asked them to reiterate their thought process to confirm they had solved them. The interviewer barely asked about their projects and only asked them to summarize those projects, possibly because they were generic projects that everyone has made. For that entry-level role, the interviewer cared more about coding skills because the company expected to train them from scratch.

Out of the things in my GitHub account there were two projects that caught the main interest of interviewers:

1. A full service written for a capillary electrophoresis device from a chemistry lab. The project required studying the data flow from the device, building a desktop application for scientists to receive and display the data stream, saving experiment results to a database, and viewing test results later in a web application. The desktop application was written in Java with JavaFX and Swing. The REST API was built with Java and Spring Boot. The database was PostgreSQL. The project was being used by scientists and lab assistants in a university chemistry building, so people were actually using what was built.

2. A web UI for remotely maintaining a web server. It could control the server over the Internet, get metrics, and send commands. It was built in Java with Spring Boot.

A third project also interested interviewers: a modded router turned into an IoT device. The router was modified to work as a router, web server, and Arduino-like board by extending pins so sensors could connect to it. The IoT project was interesting, but companies focused more on the first two projects when the role did not involve embedded programming.

The key is not some awesome project, the key is networking and good soft skills. Projects that landed me my first job were in Java: Tetris in the terminal without any GUI and a tool that could zip files, also without a GUI. The point is to show that you have some programming skills.

What lands you the actual job is setting up a good LinkedIn profile and contacting as many recruiters as you can.

During the job interview, you should leave the impression of a highly motivated person who is willing to learn and is not afraid to ask questions. The company doesn't care what you can do; they know that you will spend a lot of time on the job learning, and they need to know that they can shape you to their tech stack.

Any project that employs all four pillars of object-oriented programming, can employ multi-threading or multi-processing, reasonably defends against race conditions, livelock, and deadlock, and also interacts with a database or other processes/systems is a fairly well-rounded project.

It is also good to have it available for review on GitHub, well documented, and with thoughtful requirements. It doesn't hurt if the documentation can demonstrate a strong UML understanding either.

Demonstrating that you can write code that works is one thing. Demonstrating that you can write code that works and is readable, maintainable, adaptable, reusable, and extensible by others will put you over the top.

Writing code is easy. Outlining a problem, researching evidence-based solutions, and forming new solutions is what separates a software engineer from a coder.

As someone who had a Bachelor's in CS but no internships or experience, I made an Android app that stores words and definitions that the user inputs. I included user authentication, and the words are stored on a containerized backend using Spring Boot and Postgres DB, hosted on an Azure VM.

My advice is, rather than having a super creative idea, take your idea and extend it as much as you can to learn about and show capability with a variety of modern tools that you will likely come across in your career.

Also, write a good design or technical document outlining the project. It doesn't have to be super long, but I think it sets you apart from those who only have code to show. Writing docs like that is a big part of the job as well.

---



I think two types of projects are beneficial, and I think you need both.

A project that interests you personally shows you actually have an interest in programming beyond a paycheck. It gives you an easy topic in an interview where you can show genuine excitement or enthusiasm.

A project that highlights your skills in a certain technology or area of expertise can be simple and to the point. If you know something like Spring Boot, Firebase, or NoSQL, actually show that. It does not have to be complete or fully fleshed out; just have the basic structure so technical interviews can see what level you're at.

I had played League of Legends forever, so I decided to build a copy of websites like u.gg or op.gg. I changed the styling and interfaces but the concept was the same. I described it more like a statistical analysis tool than a video game website, and got a lot of credit for understanding and having experience in the pieces that went into it: UI development, frontend, scripting, backend, databases, etc.

By having experience with the big picture, I looked more attractive in any of those individual areas. I now work as a full-time software engineer.

Work on something you're passionate about. It will make the work easier and motivate you to dive in deeper than you would otherwise. Then find a way to make the time spent look valuable. Don't describe it as a worthless pastime, because it isn't.
